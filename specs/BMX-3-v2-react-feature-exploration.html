Spike: explore the V2 React version for feature ideas

A feature-mining report on bookmark-express-next — the earlier Manifest V2, React + Material-UI build — recommending which of its ideas to carry into the shipping product, bookmark-express-v3 (Manifest V3, Svelte + TypeScript).

Ticket: BMX-3 Phase: refinement Type: spike (exploration — no product code) Date: 2026-07-20
What this document is. A spike's deliverable is knowledge, not shipped code, so this report is the deliverable: it inventories every feature the V2 React version has, marks each against what bookmark-express-v3 already ships, and turns the gap into a prioritized, ticket-ready backlog. No app or source changes are made. A later session picks up the recommendations below as their own build tickets.

1. TL;DR

The V2 React version was a playground for bookmark management features that the current product deliberately left out when it was rewritten as a lean MV3 launcher. Four of its ideas are worth bringing back:

  • P1 — Bookmark the current page from the popup (its flagship feature; turns BMX from a read-only launcher into a capture tool).
  • P1 — Per-result actions: Delete & Reset count (cheap, high-utility housekeeping right where you already are).
  • P2 — Recently-added bookmarks as the empty-query view (fills today's blank popup with something useful; ~a dozen lines).
  • P3 — Folder favorites / recent / browse-tree picker (only needed to make "bookmark this page" pleasant; the largest UI lift, so it trails the P1 that motivates it).

Everything else the React version did is either already in v3 (search, ranking, highlighting, breadcrumbs, keyboard nav) or is an MV2/MUI artifact not worth porting (see §6).

2. Context — three codebases, one product line

The repo carries three generations of the same extension. Knowing which is which is the whole point of this spike:

DirectoryManifestStackRole
bookmark-expressV2Backbone / jQuery / RequireJSThe original, on the Chrome Web Store.
bookmark-express-nextV2React 17 + Material-UI + TSThe "V2 React version" — this spike's subject. An experiment that grew management features; never shipped.
bookmark-express-v3V3Svelte + TS on ViteThe current, actively-developed product. The target for any ideas we adopt.

The v3 rewrite was scoped as a faithful, fast MV3 port of the original — instant search and launching — and consciously dropped the management features the React experiment had added. This spike revisits that experiment to decide what, if anything, deserves a second life in v3.

3. What bookmark-express-v3 already ships

So we don't re-propose things that exist, here is the current baseline (from the v3 README and src/):

  • Instant search of Chrome bookmarks (min 2 chars), most-used ranked first, folder path per result, title+URL match highlighting.
  • Full keyboard nav (arrows / Enter); open in new vs. same tab with an invert toggle; incognito open via Cmd/Ctrl+Shift+Enter.
  • Modern command-palette look and a classic look, switchable; theme System / Light / Dark.
  • MV3 favicons via the _favicon/ endpoint; chrome.storage.local cache with FNV-1a change detection (lib/bookmarks.ts).
  • Options page for the three settings above (lib/settings.ts: invertTabBehavior, useClassic, theme).

Notably, v3's BookmarkStore is read-only — it does search() and recordAccess() and nothing that mutates bookmarks. Every recommendation below that writes is genuinely new surface area for v3.

5. Gap analysis

The React experiment's whole theme is write access — creating, deleting, and organizing bookmarks — plus the folder-navigation UI needed to support it. v3 is a pure read/launch tool. So the gap is coherent: adopting any of these means giving v3 its first mutating operations and (for the add flow) its first folder-navigation UI.

One thing already covered. The React version's "Show URLs / Show breadcrumbs" toggles are effectively on by default in v3's modern view (it shows full URLs and folder breadcrumbs already). Re-adding them as toggles is not recommended unless users ask to hide them — noted here only so it isn't mistaken for a missing feature.

6. Recommendations — a prioritized, ticket-ready backlog

Each block below is sized to become its own follow-up ticket. Effort is relative (Svelte, no Material-UI). "Slots into" names the v3 code the work extends.

P1 Bookmark the current page

Value: highEffort: mediumRisk: new write path + permissions

The React version's flagship. A "+" affordance in the popup opens a small form pre-filled with the active tab's title and url (and favicon), lets the user tweak them, choose a destination folder, and save. This is the single feature that most changes what BMX is — from a launcher into a capture-and-launch tool.

React refs: SearchBar.tsx (Add), EditBookmark.tsx. Slots into v3: new add mode in App.svelte, a new create method on BookmarkStore wrapping chrome.bookmarks.create, a new components/AddBookmark.svelte.

Acceptance criteria

  • From the popup, the user can open an "add" view that is pre-populated with the current tab's title and URL.
  • Title and URL are editable before saving; the active tab's favicon is shown.
  • Saving calls chrome.bookmarks.create into the chosen folder and closes the popup; the new bookmark appears in subsequent searches.
  • Choosing a destination folder is possible; the default target is defined (e.g. the last-used folder or "Bookmarks Bar") when the picker (P3) is not yet present.
  • Requires only the minimum permissions (bookmarks already granted; add tabs/activeTab as needed) — documented in the manifest change.

P1 Per-result actions: Delete & Reset count

Value: highEffort: lowRisk: destructive action needs a guard

A lightweight per-result menu offering Delete (remove the bookmark) and Reset count (zero its usage so it stops floating to the top). Cheap to build, and it lets users do housekeeping without leaving the popup — a natural pairing with P1.

React refs: ResultActions.tsx, Results.tsx. Slots into v3: ModernView.svelte/ClassicView.svelte result rows; new remove + reset methods on BookmarkStore (chrome.bookmarks.remove; clear timesAccessed in bookmarkData).

Acceptance criteria

  • Each result exposes an actions affordance that does not interfere with keyboard nav or the click-to-open behavior.
  • Delete removes the bookmark via chrome.bookmarks.remove and the row leaves the list (or is visibly disabled); deletion is confirmed or easily undone, since it is destructive.
  • Reset count sets that bookmark's timesAccessed to 0, persists it, and the result re-ranks accordingly on the next search.
  • Both actions work in the modern view; parity in the classic view is decided explicitly.

Simplify vs. the React original: drop the justDeleted opacity-0.2 "ghost row" hack — in v3 just remove the row (or toast with undo).

P2 Recently-added bookmarks as the empty-query view

Value: mediumEffort: lowRisk: minimal

Today v3's popup is blank until you type two characters. The React version had a History button showing the 30 most recently added bookmarks. Better than a button: make that list the default content of the empty popup, so it's useful the instant it opens and typing replaces it with search results.

React ref: SearchBar.tsx (chrome.bookmarks.getRecent(30)). Slots into v3: App.svelte renders a recent list when query.length < 2 instead of nothing.

Acceptance criteria

  • With an empty/one-char query, the popup shows the N most recently added bookmarks (N configurable in code, default ~30) using the same result row UI.
  • Typing a 2+ char query replaces the recent list with search results; clearing it restores the recent list.
  • Rows behave identically to search results (open, keyboard nav, and any P1b actions).

P3 Folder picker: favorites, recently-used & browse-tree

Value: medium (multiplier on P1)Effort: highRisk: most UI to rebuild without MUI

The React "add" flow let you target a folder three ways: starred Favorites, Recently Used folders, and a full Browse tree — with favorites managed on the Options page. This is what makes "bookmark this page" pleasant for someone with a deep folder hierarchy, but it is also the biggest lift (a tree widget and a new favorites store, all hand-built in Svelte since there is no MUI TreeView). It trails P1 deliberately: ship the add flow with a sensible default folder first, then layer this on.

React refs: FolderSelection.tsx, BrowseFolders.tsx, options-page/index.tsx. Slots into v3: a new components/FolderPicker.svelte + FolderTree.svelte; a favorites store in lib/settings.ts or a new lib/folders.ts; Options additions in Options.svelte.

Acceptance criteria

  • The add view (P1) can target any folder chosen from a browsable tree of the bookmark hierarchy.
  • Users can mark/unmark folders as favorites (persisted); favorites appear as quick-pick chips in the picker.
  • Recently-used destination folders are offered as quick picks.
  • Favorites management has a home on the Options page.
  • Consider splitting into sub-tickets: (a) browse-tree picker, (b) favorites, (c) recently-used — favorites/recent are optional polish on top of (a).

Explicitly not recommended

  • Show URLs / Show breadcrumbs toggles — v3's modern view already shows both by default; only add toggles if users ask to hide them.
  • High-res favicon fetching — the original motivation (low-res chrome://favicon/) is gone in v3, which uses the MV3 _favicon/ endpoint. Revisit only if favicon quality is a real complaint.
  • Porting anything Material-UI / React-shaped verbatim — the MUI accordions, Avatar, TreeView, and the window.open / chrome://favicon/ calls are MV2/stack artifacts. Adopt the ideas; rebuild the UI in v3's existing Svelte components and its _favicon/ + openMode conventions.

7. Risks & open questions

  • Permissions. Writing bookmarks and reading the active tab may widen the manifest (tabs/activeTab). Confirm the minimal set; a permissions bump can affect the Web Store review and the install prompt.
  • Destructive actions. Delete needs a confirm or undo; the popup closing on open makes an undo toast slightly awkward — decide the UX.
  • Scope of the write surface. Adding create/remove makes BookmarkStore stateful about the tree; ensure the cache/hash invalidation still holds after a create or delete.
  • Classic vs. modern parity. Decide up front whether new actions appear in the classic look or modern-only.
  • Should these be one epic or separate tickets? Recommendation: P1 (add) and P1b (delete/reset) ship independently and first; P2 is a quick standalone; P3 is its own epic that depends on P1.
